# ТЗ: Клубный Telegram-бот с VPN-доступом

## 1. Общее описание

Telegram-бот, работающий в привязке к клубной Telegram-группе. Бот предоставляет участникам группы возможность получить VPN-ключ (подписку) через панель 3X-UI. Авторизация основана на членстве пользователя в группе.

## 2. Глоссарий

| Термин | Описание |
|--------|----------|
| Группа | Telegram-группа (или супергруппа), к которой привязан бот |
| 3X-UI | Панель управления VPN-сервером (Xray) с REST API |
| Подписка | Клиентская конфигурация (ключ) для подключения к VPN |
| Ключ | Сгенерированная ссылка/конфигурация для VPN-клиента |

## 3. Роли и авторизация

### 3.1 Механизм авторизации

При каждом обращении к боту (любая команда) выполняется проверка:

1. Проверка, не забанен ли пользователь в боте (флаг `is_banned` в БД).
   - Если забанен — бот отклоняет запрос с сообщением: _«Вы заблокированы. Обратитесь к администратору»_. Дальнейшие проверки не выполняются.
2. Бот запрашивает у Telegram API статус пользователя в привязанной группе (`getChatMember`).
3. Если пользователь является участником группы (статус: `member`, `administrator`, `creator`) — он считается **авторизованным**.
4. Если пользователь не в группе или покинул её — бот отклоняет запрос с сообщением: _«Вы не являетесь участником клуба»_.

### 3.2 Роли

| Роль | Описание | Назначение |
|------|----------|------------|
| `USER` | Участник группы | Автоматически при первом обращении к боту |
| `ADMIN` | Администратор бота | Назначается через команду `/promote` или при первом запуске (seed-админ) |

Роль хранится в поле `role` таблицы `users` в БД.

> **Seed-админ:** при первом запуске бота, если в БД нет ни одного `ADMIN`, пользователь с `telegram_id` из переменной `SEED_ADMIN_ID` автоматически получает роль `ADMIN` при первом обращении. После этого управление ролями происходит только через команды бота.

> **Примечание:** роль `ADMIN` не зависит от статуса администратора в Telegram-группе.

## 4. Функциональные требования

### 4.1 Команда `/start`

- **Доступ:** все пользователи
- **Поведение:**
  - Проверка авторизации.
  - При успехе — приветственное сообщение с кратким описанием доступных команд.
  - При неудаче — сообщение об отказе.

### 4.2 Команда `/vpn`

- **Доступ:** авторизованные пользователи (`USER`, `ADMIN`)
- **Поведение:**
  1. Проверка авторизации.
  2. Проверка, не заблокирован ли пользователь администратором (флаг `vpn_blocked`).
     - Если заблокирован — сообщение: _«Ваш доступ к VPN заблокирован. Обратитесь к администратору»_.
  3. Проверка наличия активного ключа в БД бота.
     - Если активный ключ **существует** и не истёк — сообщение: _«У вас уже есть активный ключ. Срок действия до: {дата}»_, и повторная отправка ключа.
     - Если ключа **нет** или он **истёк**:
       a. Создание клиента (inbound client) через API 3X-UI с лимитом по времени — **7 дней**.
       b. Получение ссылки-подписки (subscription link) или конфигурации.
       c. Сохранение записи в БД бота (user_id, ключ, дата создания, дата истечения).
       d. Отправка ключа пользователю с инструкцией по подключению.

### 4.3 Команда `/status`

- **Доступ:** авторизованные пользователи (`USER`, `ADMIN`)
- **Поведение:**
  - Показывает текущий статус VPN-ключа: активен / истёк / отсутствует / заблокирован.
  - Если ключ активен — показывает дату истечения и оставшееся время.

### 4.4 Команда `/block <user_id или @username>`

- **Доступ:** `ADMIN`
- **Поведение:**
  1. Проверка авторизации и роли `ADMIN`.
  2. Установка флага `vpn_blocked = true` для указанного пользователя.
  3. Деактивация текущего ключа пользователя в 3X-UI (disable client).
  4. Подтверждение: _«Пользователь {user} заблокирован»_.

### 4.5 Команда `/unblock <user_id или @username>`

- **Доступ:** `ADMIN`
- **Поведение:**
  1. Проверка авторизации и роли `ADMIN`.
  2. Снятие флага `vpn_blocked = false` для указанного пользователя.
  3. Подтверждение: _«Пользователь {user} разблокирован»_.

### 4.6 Команда `/ban <user_id или @username>`

- **Доступ:** `ADMIN`
- **Поведение:**
  1. Проверка авторизации и роли `ADMIN`.
  2. Установка флага `is_banned = true` для указанного пользователя.
  3. Деактивация текущего VPN-ключа пользователя в 3X-UI (если есть).
  4. Подтверждение: _«Пользователь {user} забанен»_.

> **Отличие от `/block`:** бан полностью блокирует доступ ко всем функциям бота, даже если пользователь остаётся участником группы. `/block` блокирует только возможность получения VPN.

### 4.7 Команда `/unban <user_id или @username>`

- **Доступ:** `ADMIN`
- **Поведение:**
  1. Проверка авторизации и роли `ADMIN`.
  2. Снятие флага `is_banned = false` для указанного пользователя.
  3. Подтверждение: _«Пользователь {user} разбанен»_.

### 4.8 Команда `/promote <user_id или @username>`

- **Доступ:** `ADMIN`
- **Поведение:**
  1. Проверка авторизации и роли `ADMIN`.
  2. Установка `role = ADMIN` для указанного пользователя.
  3. Подтверждение: _«Пользователь {user} назначен администратором»_.
  4. Если пользователь уже `ADMIN` — сообщение: _«Пользователь {user} уже является администратором»_.

### 4.9 Команда `/users`

- **Доступ:** `ADMIN`
- **Поведение:**
  - Список пользователей, получивших VPN-ключи.
  - Для каждого: имя/username, роль, статус ключа (активен/истёк), дата истечения, заблокирован VPN или нет, забанен или нет.

## 5. Модель данных

### 5.1 Таблица `users`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | SERIAL, PK | Внутренний ID записи |
| `telegram_id` | BIGINT, UNIQUE, NOT NULL | Telegram user ID |
| `username` | VARCHAR(255), NULLABLE | Telegram username |
| `first_name` | VARCHAR(255), NULLABLE | Имя пользователя |
| `role` | VARCHAR(16), NOT NULL, DEFAULT 'USER' | Роль: `USER` или `ADMIN` |
| `vpn_blocked` | BOOLEAN, DEFAULT false | Блокировка VPN-доступа |
| `is_banned` | BOOLEAN, DEFAULT false | Полный бан в боте |
| `banned_at` | TIMESTAMPTZ, NULLABLE | Дата бана |
| `created_at` | TIMESTAMPTZ, DEFAULT now() | Дата первого обращения |
| `updated_at` | TIMESTAMPTZ, DEFAULT now() | Дата последнего обновления |

### 5.2 Таблица `vpn_keys`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | SERIAL, PK | Внутренний ID записи |
| `user_id` | INTEGER, FK → users.id | Владелец ключа |
| `xui_client_id` | VARCHAR(255), NOT NULL | ID клиента в 3X-UI |
| `subscription_url` | TEXT, NOT NULL | Ссылка подписки / ключ |
| `created_at` | TIMESTAMPTZ, DEFAULT now() | Дата создания |
| `expires_at` | TIMESTAMPTZ, NOT NULL | Дата истечения |
| `is_active` | BOOLEAN, DEFAULT true | Активен ли ключ |

### 5.3 Миграции

Миграции управляются через Prisma Migrate (`npx prisma migrate dev` / `npx prisma migrate deploy`).

## 6. Интеграция с 3X-UI

### 6.1 Аутентификация

- Бот авторизуется в 3X-UI через REST API (`POST /login`) с логином и паролем из конфигурации.
- Сессионный cookie сохраняется и обновляется при истечении.

### 6.2 Операции

| Операция | Метод API | Описание |
|----------|-----------|----------|
| Создание клиента | `POST /panel/api/inbounds/{id}/addClient` | Создание нового клиента с `expiryTime` = now + 7 дней |
| Деактивация клиента | `POST /panel/api/inbounds/{id}/updateClient` | Установка `enable: false` |
| Получение info | `GET /panel/api/inbounds/list` | Получение информации об inbound |

### 6.3 Параметры клиента

- `email` — формируется как `tg_{telegram_id}` (уникальный идентификатор)
- `expiryTime` — Unix timestamp в миллисекундах (now + 7 дней)
- `enable` — `true` при создании
- Протокол и настройки inbound определяются конфигурацией на стороне 3X-UI

## 7. Конфигурация

Переменные окружения (`.env`):

```
# Telegram
BOT_TOKEN=<токен бота>
CLUB_GROUP_ID=<ID группы>
SEED_ADMIN_ID=<telegram user ID первого администратора>

# 3X-UI
XUI_BASE_URL=<URL панели, например http://server:2053>
XUI_USERNAME=<логин>
XUI_PASSWORD=<пароль>
XUI_INBOUND_ID=<ID inbound для создания клиентов>

# PostgreSQL
DATABASE_URL=postgresql://user:password@localhost:5432/club_bot

# VPN
VPN_KEY_DURATION_DAYS=7
```

## 8. Обработка ошибок

| Ситуация | Поведение |
|----------|-----------|
| Telegram API недоступен | Логирование ошибки, сообщение пользователю: _«Сервис временно недоступен»_ |
| 3X-UI API недоступен | Логирование ошибки, сообщение: _«Не удалось создать VPN-ключ, попробуйте позже»_ |
| Пользователь не в группе | _«Вы не являетесь участником клуба»_ |
| Пользователь забанен | _«Вы заблокированы. Обратитесь к администратору»_ |
| VPN-доступ заблокирован | _«Ваш доступ к VPN заблокирован»_ |
| Ключ уже существует | _«У вас уже есть активный ключ»_ + повторная отправка ключа |
| Неизвестная команда | Игнорируется или сообщение с подсказкой `/start` |

## 9. Безопасность

- Бот общается с пользователями **только в личных сообщениях** (команды в группе игнорируются или обрабатываются с ответом в ЛС).
- VPN-ключи отправляются **только в личные сообщения** — никогда в группу.
- Пароль 3X-UI хранится только в переменных окружения, не в коде и не в БД.
- Логирование не включает чувствительные данные (пароли, ключи).

## 11. Сценарии использования

### Сценарий 1: Новый пользователь получает VPN

```
Пользователь → /start → Бот проверяет группу → ✅ в группе
                         → «Добро пожаловать! Используйте /vpn для получения ключа»
Пользователь → /vpn   → Проверка авторизации → ✅
                       → Проверка блокировки → ✅ не заблокирован
                       → Проверка ключа → ключа нет
                       → Создание клиента в 3X-UI → ✅
                       → Сохранение в БД
                       → «Ваш VPN-ключ: {ключ}. Действителен до: {дата}»
```

### Сценарий 2: Повторный запрос VPN

```
Пользователь → /vpn → Проверки пройдены
                     → Ключ существует и активен
                     → «У вас уже есть активный ключ. Действителен до: {дата}»
                     → Повторная отправка ключа
```

### Сценарий 3: Админ блокирует пользователя

```
Админ → /block @username → Проверка роли ADMIN → ✅
                          → Установка vpn_blocked = true
                          → Деактивация клиента в 3X-UI
                          → «Пользователь @username заблокирован»
```

### Сценарий 4: Пользователь покинул группу

```
Пользователь → /vpn → Проверка членства в группе → ❌ не в группе
                     → «Вы не являетесь участником клуба»
```

### Сценарий 5: Админ банит пользователя

```
Админ → /ban @username → Проверка роли ADMIN → ✅
                        → Установка is_banned = true
                        → Деактивация VPN-ключа в 3X-UI (если есть)
                        → «Пользователь @username забанен»
```

### Сценарий 6: Забаненный пользователь пытается использовать бота

```
Пользователь → /vpn → Проверка бана → ❌ забанен
                     → «Вы заблокированы. Обратитесь к администратору»
(любая другая команда — аналогично)
```

## 13. Критерии готовности (Definition of Done)

- [ ] Бот запускается и отвечает на `/start`
- [ ] Авторизация через проверку членства в группе работает корректно
- [ ] Команда `/vpn` создаёт клиента в 3X-UI и возвращает ключ
- [ ] Повторный `/vpn` возвращает существующий ключ без дублирования
- [ ] Администратор может блокировать/разблокировать VPN-доступ пользователей
- [ ] Заблокированный пользователь не может получить/использовать VPN
- [ ] Администратор может забанить/разбанить пользователя
- [ ] Забаненный пользователь не может использовать ни одну команду бота (даже если состоит в группе)
- [ ] Ключи имеют срок действия 7 дней
- [ ] Бот корректно обрабатывает ошибки API
- [ ] Администратор может назначить роль `ADMIN` через `/promote`
- [ ] Seed-админ создаётся при первом запуске
- [ ] Чувствительные данные не утекают в логи или чат
